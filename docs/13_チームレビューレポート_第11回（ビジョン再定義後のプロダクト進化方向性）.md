## 1. 現状の機能をビジョンで再評価する

**PdM**: 「まず現状の主要機能を "業者と対等になれるか" という軸で採点してみましょう」

**DS**: 「やってみる:

| 機能 | ビジョン貢献度 | 理由 |
|------|--------------|------|
| 駅別相場マップ | ◎ | エリアの価格感を一覧できる。業者が "このエリアは高い" と言ったとき検証できる |
| 取引一覧（過去成約） | ◎ | 業者が "相場はこのくらい" と言ったとき、実データで反論できる |
| 価格推移グラフ | ◎ | "今が買い時か" を客観的に判断できる |
| AI傾向分析 | ○ | エリアの傾向を把握できる。ただし "業者と戦う" より "知識を補完する" 性格 |
| AI提案コンシェルジュ | △ | 便利だが "業者に対抗する" というより "業者の代替" に近い |
| お気に入り・比較機能 | △ | UX的に良いが、業者との情報格差解消に直接つながらない |
| SUUMO外部リンク | ○ | "現在の売り出し価格" へ誘導する。相場との乖離を自分で確認できる前提で価値が出る |
| オンボーディングウィザード | △ | 予算の見積もり支援。業者に言われた予算が妥当かは別の話 |
」

**Sales**: 「面白い評価ですね。取引一覧が◎なのはわかる。 "2024年にこの駅で4,200万で成約した事例があります" という事実が、業者の言い値に対する最強の対抗材料なんですよ」

**PdM**: 「AIコンシェルジュが△というのは意外と重要な指摘かも。 "AIに選んでもらう" はビジョンとズレがある。ビジョンは "自分で判断できるようにする" であって、 "AIに決めてもらう" ではない。コンシェルジュは便利機能として残すけど、メインストーリーではないということ」

**Dev**: 「確かに。プロダクトのコアは "データを見て自分で判断できる状態にする" で、AIはその補助であるべきだな」

## 2. ビジョンから逆算すると何が足りないか

**PdM**: 「"業者と対等に" を実現するためのユーザー行動を分解すると:

1. **相場を知る** ── 今の機能で対応できている
2. **物件の価値を評価する** ── 今の機能では "相場がわかる" だけ。"目の前の物件が高いか安いか" を判定する機能がない
3. **交渉の根拠を持つ** ── 今の機能では "データを自分で見る" だけ。業者に提示できる形に加工する機能がない
4. **業者のトークを見抜く** ── 全く対応していない

2〜4が完全に空白なんですよね」

**Sales**: 「そうなんですよ。うちのお客さんがよく困るのが "業者が提示してきた価格を見て、それが適正かどうかわからない" という段階。相場マップで "浦和美園の相場は4,000万くらい" とわかっていても、業者に "この物件は4,500万です" と言われたとき、"500万高いけど交渉できる？ それとも特別な理由がある？" を判断できない」

**DS**: 「つまり、"エリアの相場" と "特定物件の価値" の間をつなぐ機能がないということ。今は前者だけを提供している」

**PdM**: 「もう一個重要な欠落があって、それは **"過去成約データ" と "現在売り出し物件" の比較** 。今は両者がバラバラに存在している。過去データはここで見て、現在物件はSUUMOへ飛んで、という分断がある。ユーザーが自分で頭の中で比較する必要がある」

**Sales**: 「現場で言うと、買い手が "SUUMO に出ている4,800万の物件を見て、ここ（相場マップ）で過去成約が4,200万だと確認して、600万交渉しようとした" という話は実際にある。でも今のUIではそれをサポートする動線がない」

## 3. 次のフェーズで作るべきもの: 「物件査定補助」機能

**PdM**: 「ここで大きな新機能提案をしたい。名前は仮で **"物件査定補助"** 。ユーザーが見ている物件情報（面積・築年数・徒歩分数・価格）を入力または貼り付けると、その駅の過去成約データと照らし合わせて "この物件は相場比 +X% / -X%" を返す機能」

**DS**: 「実現方法を考えると: ①ユーザーが物件スペックを入力（面積・築年数・駅徒歩・価格） → ②その駅の `transactions/*.json` から同条件（面積±10㎡、築年数±5年、徒歩±3分）の成約事例を抽出 → ③中央値と比較してパーセンテージ差を計算 → ④AIが "相場より高い理由として考えられること / 交渉余地のある根拠" を文章生成」

**Dev**: 「データは全部手元にある。取引JSONは全175駅分が `public/transactions/` に存在する。フロントエンドだけで完結できるし、AI生成部分は既存の `/api/summary` の応用でできる。実装コストは中程度だな」

**Sales**: 「これは強い。"この物件は2024年の同条件成約事例（平均4,250万円）に対して12%高い価格設定です" と出てきたら、業者に "成約データで確認したので、4,300万まで下げてもらえますか" と言える。完全に業者と対等な交渉ができる」

**PdM**: 「UIイメージとしては、Screen 2（駅詳細ページ）の中に "物件チェック" セクションを設ける。物件スペックをフォームで入力すると、その場で比較結果が表示される」

**Dev**: 「一つ確認したいのが、SUUMO などで見た物件を "URLを貼るだけで自動入力" できたら UX が劇的に良くなるけど、これはスクレイピングになる。手動入力形式が現実的だな」

**DS**: 「手動入力でも十分価値はある。面積・築年数・徒歩・価格の4項目を入力するだけなら、ユーザーも苦にならない」

**PdM**: 「出力イメージは？」

**Sales**: 「こんな感じ:

---
**〇〇駅 物件査定レポート（参考）**

■ 対象物件: 〇〇駅 徒歩8分 / 75㎡ / 築18年 / 4,800万円
■ 過去成約データとの比較（2023〜2025年、同条件）:
　- 類似成約件数: 12件

これを持って業者に行けば、ちゃんと交渉できる」

**Dev**: 「テキストとして表示するだけなら実装は簡単。コピーボタンをつければ実用的。PDFエクスポートは後回しでいい」

**PdM**: 「"国土交通省のデータに基づく" という出典明記は、信頼性担保として重要。公的データであることがプロダクトの権威性になる」

## 5. 「買い時スコア」の可能性

**DS**: 「もう一個、ビジョンに沿った機能案を出したい。 **"買い時スコア"** 。"このエリアは今買い時か" をスコア化する。価格推移の傾向（上昇/横ばい/下落）と取引件数の変化から、簡易的なスコアを算出できる」

**PdM**: 「業者はよく "今が買い時です！" と言う。それが本当かどうかをデータで検証できるということですね」

**DS**: 「計算ロジック案: ①直近2年の価格変化率（上昇率が高い → 買い時スコア低い、横ばい → 中、下落傾向 → 高い） ②取引件数の変化（減少 → 供給過多の可能性） ③価格の全国/埼玉平均との乖離。これらを組み合わせた0〜100のスコアで表示。"このエリアの買い時スコア: 68点（比較的安定した価格水準）" みたいな感じ」

**Sales**: 「業者がよく使う "今がお買い得です" という言葉への反証ツールになる。データ的に "今が特別割安なわけではない" を示せれば、焦らせる営業トークに乗らなくて済む」

**Dev**: 「既存の `station.years` データから計算できる。新しいデータ収集は不要。ただ "スコアの定義" が難しい。スコアが一人歩きして "スコアが低いから買わない方がいい" と誤解されるリスクがある」

**PdM**: 「スコアよりも "傾向表示" の方が安全かも。 "直近2年: 価格 +4.2% / 取引件数 -8%" という数値をそのまま見せる。判断はユーザーに委ねる。"業者と対等に" というビジョンに照らすと、 "スコアで誘導する" より "データを渡して自分で判断させる" 方が哲学的に一貫している」

**DS**: 「その通りだな。過剰に意思決定を代行しない。データを渡す存在としてのポジションを守る」

## 10. フェーズの再定義

**PdM**: 「今後のフェーズを、ビジョン "業者と対等に" の達成度で再定義したい」

```
Phase A: 相場を知る（✅ 完了）
  → エリアの価格水準・推移を把握できる状態

Phase B: 個別物件を評価する（← 次のフォーカス）
  → 目の前の物件が相場と比べて高いか安いかを判断できる状態
  → K-03（物件査定補助）・K-04（交渉レポート）

Phase C: 業者と交渉できる（Phase B の先）
  → 根拠を持って価格交渉に臨める状態
  → 交渉レポートの質向上・業者トーク対策コンテンツ

Phase D: 全プロセスを支援する（将来）
  → エリア選定〜交渉〜契約まで一貫して情報格差を埋める
  → 関東全域への拡張・エリアスコアリング・利回り計算
```

**Sales**: 「このフレームワーク、シンプルで良い。Phase A が終わって Phase B に進む、という説明は外部にも伝わりやすい」

**Dev**: 「Phase B の実装難易度を確認すると: 取引データは揃っている、比較計算はフロントエンドで完結できる、AI生成は既存インフラを流用できる。Phase A よりも実は難易度は低いかもしれない。問題は "計算ロジックの設計（何を同条件とみなすか）" だな」

**DS**: 「同条件の定義が肝心。面積は ±15㎡、築年数は ±7年、徒歩は ±3分くらいが現実的な範囲かな。マッチする件数が少なすぎる（5件未満）場合は "比較可能なデータが不足しています" と表示するのが誠実」
