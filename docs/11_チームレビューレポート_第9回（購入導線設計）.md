## 2. 現状レビュー: 既存の SUUMO リンク実装

**Dev**: 「現在の実装を確認すると、Screen 2 の末尾に "現在の売り出し物件を探す" セクションがある。SUUMO へのリンクが1つだけで、URL は `?kw={駅名}` の駅名フリーワード検索。ボタンデザインは緑背景の `bg-green-50` で外部リンクアイコン付き」

**Sales**: 「実際にリンクを叩いてみると… 駅名で検索されるんですよね。これ、正直微妙で。"浦和" で検索すると浦和駅のマンション一覧は出るんですが、予算フィルターがかかってない。4,000万予算で相場を調べたユーザーが SUUMO に飛んで8,000万の物件が並んでいたら混乱する」

**PdM**: 「なるほど。相場マップで "自分の予算で買えるかどうか" を確認してから外部リンクに飛ぶのに、その予算情報が引き継がれていないのは確かにもったいない」

**DS**: 「SUUMO の URL パラメータは価格帯指定できるはず。`?cn=9999&lowprice=0&highprice=5000` みたいな形で。実際どういう構造になってるか確認が必要だけど、`budgetMax` を URL に含めることは技術的には難しくない」

**Dev**: 「SUUMO の URL パラメータは公式ドキュメントがないので、ブラウザで操作した URL を解析するしかない。ただ、万単位の価格パラメータを渡せることは確認してる。エリアコード（`ar=030` は関東）と物件種別コード（`bs=011` は中古マンション）は今も正しく設定されてる」

**Sales**: 「もう一点。SUUMO だけだとリーチが偏ります。アットホームは地場の中小不動産会社の掲載が多くて、SUUMO にない物件が結構ある。特に埼玉の外周部は地場業者の物件が多いので、両方リンクがあると親切」

## 3. 購入導線の設計原則: ユーザーの頭の中を整理する

**PdM**: 「ここで一度、ユーザーがこのプロダクトをどう使っているかを整理したい。仮説ベースで3タイプのユーザーがいると思う:

- **タイプA（情報収集中）**: "埼玉ってだいたいどのくらいか知りたい" 。まだ買う気は固まってない。相場マップでエリア感を掴む段階
- **タイプB（エリア検討中）**: "与野本町か浦和美園か迷ってる" 。特定の駅を比較している段階
- **タイプC（購入直前）**: "浦和美園で4,000万以内の3LDKを探している" 。実際の物件を探したい段階

外部リンクに価値があるのは主にタイプCで、タイプAにとっては離脱リスクになりかねない」

**Sales**: 「的確な整理ですね。現場的に言うと、タイプBとCの間に "このエリアで本当に買えるのか？" という不安確認の段階がある。取引一覧を見て、予算内の過去成約を確認することで "買えた人がいる" と安心する。その次にはじめて "今売ってる物件を見たい" になる」

**DS**: 「Screen 2 の情報構成がその流れに沿ってるのは良い。価格サマリ → AI傾向分析 → 取引一覧（過去成約）→ 外部リンク（現在物件）の順序は、ユーザーの心理ステップと合致してる」

**PdM**: 「そうすると購入導線の強化策は2つの方向がある: ① 外部リンクの質を上げる（より的確な条件で飛ばす）、② 実際の物件情報を一部取り込む（サイト内で完結させる）。コスト・効果を踏まえてどちらを優先するか議論しよう」

## 4. 方針A: 外部リンク強化（低コスト・高効果）

**Dev**: 「外部リンク強化の実装案は以下:

1. **予算付きSUUMO URL**: `budgetMax` を URL パラメータで渡す。`highprice` パラメータに万円単位で設定
2. **アットホームリンク追加**: `https://www.athome.co.jp/mansion/chuko/saitama/{駅名}/` 形式で遷移できるか検証が必要
3. **HOMES（ライフルホームズ）リンク**: `https://www.homes.co.jp/mansion/chuko/saitama/station-{コード}/` — ただし駅コードマッピングが必要
4. **間取りフィルターの引き継ぎ**: `targetArea` から間取りを推定して URL に含める（例: 70㎡ → 3LDK 相当）」

**Sales**: 「優先順位をつけると、① 予算引き継ぎが断然最優先。② アットホームを追加。③ HOMES は後でいい。間取り引き継ぎは SUUMO の URL 構造的に難しければスキップでいい」

**PdM**: 「予算引き継ぎの UX を考えると、リンクボタン上に "予算4,000万円以下で絞り込んで表示" みたいなテキストがあると、ユーザーが "自分の条件で検索してくれる" と信頼感が上がる」

**Dev**: 「ボタンを2段構成にする案: 上段に "SUUMO で探す（予算4,000万以下）"、下段に "アットホームで探す" みたいな形。予算なしの場合は "条件なしで検索" に変わる」

**DS**: 「一点確認。SUUMO の URL にパラメータを付けても、実際にフィルターが効いた状態で表示されるか保証が必要。たまに URL 構造が変わって古いパラメータが無効になることがある。リリース前に実際のブラウザで確認必須」

## 5. 方針B: 物件情報の取り込み（中コスト・高効果だが制約あり）

**PdM**: 「実際の現在物件情報をサイト内に表示することは技術的にできるか？」

**Dev**: 「いくつか選択肢がある:

① **スクレイピング**: 法的リスクあり・利用規約違反になりうる → 採用不可
② **公式API**: SUUMO は非公開API。アットホームはAPIを提供しているが法人契約が必要
③ **アフィリエイトウィジェット**: SUUMO・アットホームがウィジェットを提供しているか調査が必要
④ **iframe埋め込み**: 外部サイトの X-Frame-Options 制限で基本的に不可
⑤ **国交省レインズ系**: 公的データだが個別物件情報は非公開

現実的には、公式提携なしで実物件情報をサイト内表示するのは難しい」

**Sales**: 「実務で言うと、不動産ポータルの API を利用するには仲介業者資格（宅建業者）または提携審査が必要なことが多いです。個人プロジェクトで通るのは難しい。ミニマムでは外部リンクが現実的」

**PdM**: 「じゃあ方針B は短期では現実的でないということですね。ただ将来の可能性として記録しておく価値はある。アフィリエイト経由での物件表示くらいなら比較的敷居が低いかも」

**DS**: 「アフィリエイト観点で言うと、SUUMO と アットホーム はアフィリエイトプログラムを持っています。資料請求や問い合わせのコンバージョンに連動する報酬形式。PV が増えれば収益化の道にもなる」

**Sales**: 「アフィリエイトリンクにするだけで、ユーザー体験は変わらず、かつ収益化の準備ができる。リンクの張り方を一手間変えるだけなので、やっておいて損はない」

## 6. 購入導線の深化: "背中を押す" UX 設計

**PdM**: 「外部リンクの質を上げるだけでなく、"この駅で買おう" と背中を押すためのUX改善も考えたい。取引一覧の活用という観点で」

**Sales**: 「過去成約データが取引一覧にあって、例えば "2024年Q3に3,800万で3LDK 72㎡が成約している" と見えると、 "今4,000万で出てる物件は相場より高いな" とか "この間取りなら買える" みたいな判断ができる。これが実は一番強い購入背中押しになる」

**PdM**: 「取引一覧に "この条件で探す →" という SUUMO 連動ボタンを行ごとに付けるのはどうか？ 例えば取引行をクリックすると、その面積・築年数・価格帯で SUUMO 検索が開く」

**Dev**: 「実装的には可能。各取引の `area`・`age`（築年数から竣工年算出）・`price` を元に SUUMO の URL パラメータを組み立てる。ただし SUUMO の建築年パラメータが竣工年基準か築年基準か確認が必要」

**DS**: 「それは "比較の基準" として非常に価値がある。 "過去に成約した類似物件" があることで、今の売り出し物件の割高・割安が判断しやすくなる。これは単純な外部リンクよりずっと深い体験」

**Sales**: 「ユーザーが実際にどうするかというと、取引一覧で "こういう物件が4,000万以内で成約していた" と確認 → SUUMO で今の物件を見る → 比較して交渉材料にする、という流れ。この流れを 1クリックで繋げられるのは大きい」

## 7. 競合・ポジショニング整理

**DS**: 「改めてこのプロダクトのポジションを確認すると: "過去成約データに基づくエリア相場マップ" は SUUMO や アットホームが持っていない視点。ポータルは "今売ってる物件" が中心で、過去の成約トレンドはあっても駅単位の詳細分析は弱い」

**PdM**: 「だから競合しているのではなく、補完関係にある。 "このプロダクトで相場を理解してから → SUUMO・アットホームで物件を探す" という導線は自然で、ポータルにとっても送客元として歓迎されるはず」

**Sales**: 「現場でも "相場感がない状態でSUUMOを見ても何が安いか高いかわからない" という声はよく聞きます。このプロダクトで相場観を作ってから物件サイトへ、という使い方はとても自然」

**Dev**: 「"相場を理解するプロダクト" と "物件を探すプロダクト" の役割分担が明確なら、外部リンクを積極的にプッシュする UI 設計も整合性が取れる。 "次のステップはSUUMOで探してみましょう" みたいなコンテキストが添えられると良い」

## 8. 具体的な実装優先順位

**PdM**: 「整理すると以下の優先順位が見えてきた:

**最優先（Screen 2 の改善・低コスト）**
1. SUUMO リンクに予算フィルターを引き継ぐ
2. アットホームリンクを追加（2サイト体制）
3. "なぜ外部リンクなのか" の一言説明を UI に追加（"相場の確認はここで、物件探しは外部サイトへ"）

**中優先（UX 深化・中コスト）**
4. 取引一覧行から "この条件で今の物件を探す →" ボタン
5. 面積・築年数のフィルター状態を外部リンクに引き継ぐ
6. アフィリエイトリンク化（収益化準備）

**低優先（将来・高コスト）**
7. 不動産ポータルとの公式 API 提携交渉
8. 物件情報のウィジェット埋め込み（提携後）」

**Sales**: 「1〜3 は明日にでもやってほしいくらいの優先度。4・5 は使い込まれてきたら追加で十分。6 は早めにやっておくと後で楽なので中優先でいい」

**Dev**: 「1〜3 は実装コストが低い。SUUMO の URL パラメータを変えるだけ。アットホームの URL 構造だけ調査が必要」

**DS**: 「4 の "取引行から外部検索" は UX 的に最も差別化になる機能。投資家・比較検討層に特に刺さる。実装は中程度のコストだけど、価値は高い」

## 10. 総評

**Dev**: 「Screen 2 に SUUMO リンクが入っていることは大きな前進。次は "リンクの質" を上げる段階で、予算引き継ぎとアットホーム追加は実装コストが低いので早めに対応すべきだな。取引行からの "この条件で探す" は差別化になる機能なので、ユーザーテストで手ごたえを確認してから実装する順序でいいと思う」

**PdM**: 「購入導線の考え方として、"相場マップ → 詳細ページ → 外部ポータルへ送客" という流れを、ユーザーが自然に歩めるように設計するのが大事。各ステップで "次にすることが明確" な UI にすることが優先度高いですね。I-03 の "なぜここ？" 案内はシンプルだけど重要。ユーザーが "過去データ" と "現在物件" を混同しないようにする配慮」

**DS**: 「データ観点での補足: 取引一覧は過去の "成約価格" であって、現在の "売り出し価格" ではない。この区別がユーザーに伝わっていないと、 "4,000万で過去成約があったから今も4,000万で買えるはず" という誤解が生まれる。SUUMOへ飛んだら現在の売り出し価格が表示されて、乖離があるときに混乱する可能性がある。I-03 の案内はそのリセット機能も担う」

**Sales**: 「正直、最初の SUUMO リンク追加（H-04）ができているだけで、他のマーケット情報サービスより一歩進んだ体験になっています。予算引き継ぎ（I-01）と アットホーム追加（I-02）で、"実際に使える" レベルになる。取引行からの個別検索（I-05）は、投資家や物件をシビアに比較する人には刺さる機能。どういうユーザーが集まってくるか見てから追加判断でいいと思いますよ」
