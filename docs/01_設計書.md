## 3. データモデル（v2）

### 3.1 駅別集計データ（stations.ts） — マップ表示用

面積換算を動的に行うため、**㎡単価ベース**に変更する。

```typescript
interface PriceStats {
  count: number;
  avgUnitPrice: number;    // 平均㎡単価（万円/㎡）
  medianUnitPrice: number; // 中央値㎡単価（万円/㎡）
}

interface StationYearData {
  all: PriceStats;
  age_0_10: PriceStats;
  age_11_20: PriceStats;
  age_21_30: PriceStats;
  age_31_plus: PriceStats;
}

interface StationData {
  stationCode: string;
  stationName: string;
  lat: number;
  lng: number;
  lines: string[];
  area: string;
  propertyType: "mansion" | "house"; // NEW: 物件種別
  years: Record<string, StationYearData>;
}
```

**表示時の換算**: `displayPrice = medianUnitPrice * targetArea`（例: 70㎡なら ×70）

これにより、面積セレクタ（50/60/70/80/90㎡）の切り替えをクライアントサイドで即座に実現できる。

### 3.2 個別取引データ（transactions/） — 詳細表示用 ← NEW

駅別に分割したJSONファイル。駅クリック時にオンデマンド読み込み。

```typescript
interface Transaction {
  id: string;
  price: number;         // 取引価格（万円）
  area: number;          // 面積（㎡）
  unitPrice: number;     // ㎡単価（万円）
  buildingYear: number | null;
  age: number | null;    // 築年数
  floorPlan: string;     // 間取り
  structure: string;     // RC, SRC, etc.
  district: string;      // 地区名
  period: string;        // 取引時期
  walkMinutes: number | null; // 徒歩分数
}

// ファイル: public/transactions/{stationCode}.json
```

**ファイルサイズ見積もり**: 148駅 × 平均50件 × 200bytes ≒ 1.5MB（分割なので1リクエスト10KB程度）

## 5. AI機能設計（Phase 3）

### 5A. AIエリア傾向分析

駅を選んだときに、その駅の取引データをもとにAIが傾向を分析する。

#### アーキテクチャ

```
[駅クリック] → [取引データ収集] → [API Route: /api/summary]
                                         │
                                    [プロンプト生成]
                                         │
                                    [LLM API呼び出し]
                                         │
                                    [サマリテキスト返却]
```

#### プロンプト設計（案）

```
あなたは不動産市場アナリストです。
以下は{駅名}駅周辺の中古マンション取引データ（{期間}）です。

[取引データサマリ]
- 全体: {件数}件, 70㎡換算中央値 {価格}万円
- 築10年以内: ...
- 築11-20年: ...
...

このデータから、以下の観点で200字程度の傾向分析を生成してください:
1. 価格帯の特徴（高い/安い/標準的）
2. 築年数による価格差の傾向
3. 周辺エリアとの比較での位置づけ
4. この駅で物件を探す人へのアドバイス
```

### 5B. AIエリア提案（物件コンシェルジュ）← NEW

ユーザーが自然言語で「こんな物件が欲しい」と入力すると、
AIが条件を解析 → データ照合 → 合致するエリア・駅を提案する。

#### ユーザーストーリー

> 「3,000万円以下で築15年以内、3LDK以上、駅徒歩10分以内のマンションが欲しい」
> → AIが条件を理解し、該当する駅を3〜5つ提案してくれる

#### アーキテクチャ

```
[ユーザー入力 (自然言語)]
        │
        ▼
[API Route: /api/suggest]
        │
   ┌────┴─────┐
   │          │
   ▼          ▼
[LLM]     [データ検索]

   │          │

        │
        ▼

        │

#### 処理フロー

```
Step 1: 条件解析（LLM）
  入力: "3000万以下で築15年以内、3LDK、駅近がいい"
  出力（structured）: {
    maxPrice: 3000,
    maxAge: 15,
    minFloorPlan: "3LDK",
    maxWalkMinutes: 10,
    propertyType: "mansion",
    otherPreferences: ["駅近"]
  }

Step 2: データ照合（サーバーサイド）
  - stations.ts の全駅データから条件に合う駅を絞り込み

  - 間取り条件 → transactions.json があれば照合

Step 3: 結果生成（LLM）
  条件マッチした駅リスト + データを渡し、
  「なぜこの駅がおすすめか」の解説テキストを生成

Step 4: レスポンス
  {
    stations: [
      { name: "蕨", reason: "3000万以下で...", matchScore: 0.92 },
      { name: "南越谷", reason: "築10-15年が...", matchScore: 0.85 },
      ...
    ],
    summary: "ご希望の条件では、JR京浜東北線の...",

### 6.2 予算算出ロジック

リバベルと同様に、一般的なローン計算の簡易モデルを採用する。

```typescript
interface BudgetInput {
  income: number;      // 年収（万円）
  assets: number;      // 資産（万円）
}

interface BudgetResult {
  estimate: number;    // 推定予算（万円） = 中央値
  min: number;         // 下限
  max: number;         // 上限
  loanLimit: number;   // ローン上限
  downPayment: number; // 頭金目安
}

function calculateBudget(input: BudgetInput): BudgetResult {
  const loanMin = input.income * 5;
  const loanMax = input.income * 8;
  const loanMid = input.income * 6.5;
  const dpMin = input.assets * 0.10;
  const dpMax = input.assets * 0.20;
  const dpMid = input.assets * 0.15;
  return {
    estimate: Math.round((loanMid + dpMid) / 100) * 100,
    min: Math.round((loanMin + dpMin) / 100) * 100,
    max: Math.round((loanMax + dpMax) / 100) * 100,
    loanLimit: loanMid,
    downPayment: dpMid,
  };
}
```
