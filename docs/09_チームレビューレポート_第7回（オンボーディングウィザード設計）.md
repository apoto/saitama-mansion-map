## 2. 提案されたウィザードの概要整理

**PdM**: 「今回の提案を整理すると、初回訪問時に3ステップのウィザードが表示される。STEP 1: 年収と資産（頭金）を入力 → 予算シミュレーターに回答済みの状態になる。STEP 2: 何人で住むか選択（単身 / カップル・夫婦 / ファミリー）→ 面積フィルターが自動設定される。STEP 3: 住みたいエリアの希望を自由入力 → AIエリア提案が自動実行されてマップが絞り込まれた状態で表示される」

**Dev**: 「実装観点から言うと、STEP 1は既存の`BudgetPanel`の入力ロジックを再利用できる。STEP 2は`filter.targetArea`の設定だから、選択肢に応じてどのtargetAreaを設定するかのマッピングが要る。STEP 3は`SuggestPanel`の入力フォームと`/api/suggest`エンドポイントを再利用する形になるな」

## 3. STEP 1（年収・資産入力）の要件詳細化

**PdM**: 「BudgetPanelの現在の実装を確認したいんだけど、年収と資産はどんな入力形式になってる？」

**Dev**: 「BudgetPanelは実装済みで、年収・資産をボタン選択式で入力する。ローン上限は年収×5〜8倍 + 頭金（資産×10〜20%）で計算する方式。計算結果の中央値が`budgetMax`になる。具体的な年収行（400/500/600/700/800/1000/1200/1500万）と資産列（なし/300/500/1000/1500/2000/3000/5000万）のマトリクス表があるはず」

**Sales**: 「ボタン選択式は操作しやすくていい。フリー入力より誤操作が少ない。ただ、年収を正直に入力してくれるかが心配で…」

**PdM**: 「それはどういう意味？」

**Sales**: 「お客さんって年収を盛って考えることが多いんですよ。"年収600万あれば6,000万借りられる" みたいな感じで過剰に試算する。ウィザードで入力する段階で "あくまで目安です" というメッセージを添えておかないと、後でギャップが出る」

**DS**: 「入力値の信頼性の問題は確かにある。でもこのツール自体が "相場観を掴む" 用途なので、精緻な計算じゃなく探索のきっかけになればいい。免責文言は必要だと思う」

**PdM**: 「STEP 1の要件確定: ①年収・資産はボタン選択式（既存BudgetPanelの選択UIを流用）②計算結果は "目安の予算" として表示③"この金額はあくまで目安です" という注意書きを一言入れる④完了後、`filter.budgetMax`が設定済みになり、ウィザードがSTEP 2に進む」

**Dev**: 「BudgetPanelからUI部分だけを切り出してウィザードの1ステップとして組み込む形にしよう。BudgetPanel自体も残しておいて、ヘッダーから再設定できる」

## 4. STEP 2（世帯人数→面積フィルター）の要件詳細化

**PdM**: 「STEP 2は "単身 / カップル・夫婦 / ファミリー" の3択で、それぞれ面積フィルターが設定される。面積のマッピングをどうするか議論したい」

**Dev**: 「現在の面積フィルターは `TARGET_AREAS = [50, 60, 70, 80, 90]`（㎡）で、デフォルトは70㎡。単身なら50〜60㎡、カップル・夫婦なら70㎡、ファミリーなら80〜90㎡あたりが自然か」

**Sales**: 「面積のイメージ感で言うと、単身は40〜55㎡、カップル・夫婦は55〜75㎡、ファミリー（子あり）は75㎡以上、っていうのが現場感覚ですね。ただ今のデータは70㎡換算での表示なので、そこにどう合わせるかが問題で」

**DS**: 「あー、そこ重要なポイントだな。`filter.targetArea`は "70㎡換算で見るか、30㎡換算で見るか" という表示調整パラメータ。FilterPanelを見るとすべての価格が`price70 / 70 * targetArea`で換算表示される。つまりSTEP 2の選択は "どの面積換算で価格を見せるか" の設定に過ぎない」

**PdM**: 「それは把握してる。ただユーザー体験として "あなたは単身だから30㎡で考えます" という文脈で設定されると、"自分のライフスタイルに合った価格感" が見えるようになる。単身者が70㎡換算の価格を見てもイメージしにくい、という問題の解消」

**Sales**: 「その通り。単身の人に "70㎡換算3,500万" と言っても "でも自分が欲しいのは30㎡の1K" ってなる。最初から30㎡換算で見せるほうが現実感がある」

**Dev**: 「マッピングの設計案を出すと：単身 → `targetArea: 30`、カップル・夫婦 → `targetArea: 60`、ファミリー → `targetArea: 80`。これでいい？」

**PdM**: 「ファミリーの80㎡は妥当だと思う。90㎡にするか迷うところだけど、子ども1人の場合は80㎡でリアル。あとでフィルターで調整できるから初期値80で進もう」

**Sales**: 「単身30㎡は1K/1Rのイメージで正しい。2人暮らし60㎡も2LDK想定として合ってる」

**DS**: 「ウィザードはあくまで初期設定だから、後でFilterPanelで調整できることが前提でいい。30 / 60 / 80 のマッピングで確定しよう」

**PdM**: 「STEP 2の要件確定: ①選択肢は単身 / カップル・夫婦 / ファミリーの3択のシンプルなカードUI②マッピング: 単身→30㎡、カップル・夫婦→60㎡、ファミリー→80㎡③選択後は "この設定はあとでフィルターパネルから変更できます" という一言を表示④完了後、`filter.targetArea`が設定済みになりSTEP 3に進む」

**Dev**: 「カードUIはシンプルにアイコン（絵文字）+ ラベルのボタンで行けるな。🧑単身 / 👫カップル・夫婦 / 👨‍👩‍👧ファミリーみたいな感じ」

## 5. STEP 3（希望エリア自由入力→AIエリア提案）の要件詳細化

**PdM**: 「STEP 3は最も議論が必要なステップ。"住みたいエリアの希望を自由入力" とのことだけど、具体的にどういう入力を想定してる？」

**Sales**: 「現場で聞かれること1位は "この予算でどこに住めますか？" だけど、もう少し条件がある場合は "都内通勤で埼玉ならどこが安いか" "大宮から近くて静かなところ" みたいな感じ。ざっくりしたエリア感覚を言葉にしたもの」

**DS**: 「既存の`SuggestPanel`の入力欄はどんな形式？」

**Dev**: 「SuggestPanelは自由テキスト入力で `/api/suggest` に投げる。プロンプトで条件解析 → 全駅データをフィルタ・スコアリング → おすすめ駅と理由を返す。入力例として "大宮に近くて4000万以内の駅が知りたい" みたいなサジェストテキストが入ってる」

**PdM**: 「ウィザードのSTEP 3では、STEP 1で設定した予算がすでに条件として入っているはず。だからユーザーが入力するのは "エリア感" や "通勤先" "生活スタイル" の希望だけでいい。予算は自動でプロンプトに組み込まれる」

**DS**: 「それ重要な設計判断だな。STEP 3の入力テキスト + STEP 1の予算 + STEP 2の面積 → `/api/suggest`に投げて、AI提案を表示する、という流れ。ウィザード完了時点でマップが絞り込まれた状態になる」

**Sales**: 「でも "エリア希望なし" もあるじゃないですか。自分の場合は "とにかく安いところ" とか "どこでもいい" という方も多い。スキップできるようにしないと」

**PdM**: 「STEP 3はオプションにしよう。"スキップする（後で試す）" ボタンを置いて、スキップした場合はAI提案なしでマップを表示する。ただしスキップしたあともヘッダーのAI提案ボタンから後で入力できる」

**Dev**: 「スキップした場合の挙動を決めておく必要があるな。ウィザード完了後、STEP 1とSTEP 2で設定した値（budgetMax + targetArea）だけが適用された状態でマップが開く、という理解でいい？」

**PdM**: 「それでいい。STEP 3の要件確定: ①自由テキスト入力欄（プレースホルダー: "例: 大宮に近くて静かなエリア・都内通勤しやすい駅など"）②STEP 1で設定した予算を自動でプロンプトに付加（ユーザーには "予算〇万円の条件で提案します" と表示）③"スキップ" ボタンあり④入力→送信後、AIが提案を生成してウィザードが閉じる、同時にマップが絞り込まれた状態（highlightedStationsが設定済み）で表示される」

**DS**: 「API呼び出しが完了するまでの待ち時間はどうする？ウィザードをローディング状態にするか、ウィザードを閉じてマップ上でローディング表示するか」

**Dev**: 「ウィザード上でローディングを表示して、完了したらウィザードを閉じる方が自然だと思う。"提案を生成中..." というスピナーを出す」

**PdM**: 「API失敗時のフォールバックも決めておこう。エラーの場合はウィザードを閉じて、SuggestPanelのエラー状態と同じ扱いにする。ウィザードが完了した状態（予算と面積は設定済み）でマップを開く」

## 6. ウィザードのUI/UX設計

**PdM**: 「全体の表示形式を決めよう。フルスクリーンのモーダルか、カード型のオーバーレイか、ページ切り替え型か」

**Dev**: 「実装コスト的にはモーダル一択。`BudgetPanel`や`SuggestPanel`と同じ `fixed inset-0 z-50` のパターン。ただフルスクリーンにするとSP対応が楽になる」

**PdM**: 「フルスクリーンにしよう。理由は2つ: ①初回ユーザーにとって "何かが始まった感" を演出できる、②SPで途中のフォームが見切れるのを防げる。デザインはシンプルに、ステップインジケーター（●○○）＋質問文＋選択UI＋"次へ/完了"ボタンの縦1カラムレイアウト」

**Sales**: 「ステップインジケーターは大事ですよ。"あと2ステップで終わる" がわかると離脱しにくくなる。不動産申し込みフォームも同じ理屈で」

**Dev**: 「アニメーションはどうする？ステップ切り替えのトランジション」

**PdM**: 「Framer Motionはすでにプロジェクトに入ってる？」

**Dev**: 「入ってないと思う。Tailwindのtransitionクラスで対応するなら、CSS only で `opacity + translate` のフェードスライドができる。Framer Motionは依存追加になるから、最初はCSSだけでいい」

**DS**: 「合意。最初はシンプルに。アニメーションは後でUX改善のフェーズで磨けばいい」

## 7. ウィザードの表示条件・永続化

**PdM**: 「ウィザードをいつ表示するか、いつ表示しないかを決めよう」

**Dev**: 「表示条件: ①初回訪問時（localStorageに完了フラグがない）②URLに`?onboarding=1`パラメータがある場合は強制表示（テスト用）。非表示条件: localStorageに`onboarding_completed`フラグがある。ウィザード完了またはスキップで`onboarding_completed: "1"`を保存」

**PdM**: 「完了したあとも再挑戦したいユーザーへの配慮はどうする？」

**Sales**: 「ヘッダーに "再設定" ボタンでも置けばいいんじゃないですか。普段は小さく "条件を最初から設定" みたいなリンクでいい」

**Dev**: 「ヘッダーが今でも4ボタン並んでいて、これ以上ボタンを増やすとごちゃつく。既存の "予算確認" ボタンをクリックしたらBudgetPanelが開くので、それで十分では。ウィザード再実行のユースケースはたぶんニッチ」

**PdM**: 「同意。ウィザードの再実行はこのフェーズでは提供しない。localStorageを手動でクリアするか、再読み込みでは再表示しない運用で」

**DS**: 「あと、URLパラメータで `?budget=XXXX` が入ってる状態でアクセスした場合（共有リンク等）はウィザードを表示しないようにすべきだな。budgetMaxがすでに設定されてるなら、もうSTEP 1は完了した状態といえる」

**Dev**: 「それは`getInitialBudget()`が`null`でない場合にウィザードをスキップする条件を追加すればいい。今の`page.tsx`の`getInitialBudget()`はURLからbudgetを読んでいるから、その値をウィザード表示判定にも使える」

## 8. 技術的懸念事項・積み残し

**Dev**: 「STEP 3のAI提案生成はウィザード内で行うため、`SuggestPanel`のfetchロジックを切り出してhookにする必要がある。今は`SuggestPanel`コンポーネント内にfetchが書かれてる形だと思うけど、ウィザードから呼ぶためには共通化が要る」

**PdM**: 「`useSuggest`みたいなカスタムhookを切り出す形？」

**Dev**: 「そう。あるいはOnboardingWizardがSuggestPanelを内部でrendering するけど非表示にして、完了したら表示するパターン。コンポーネントの再利用は難しいことが多いから、hookを切り出す方が綺麗かな」

**DS**: 「STEP 3で生成されたAI提案の結果は、ウィザード閉幕後も`SuggestPanel`側に引き継がれるべきか？そうじゃないと、ウィザードでAI提案が出てマップが絞り込まれるのに、後でSuggestPanelを開いたら空になる、という状態になる」

**Dev**: 「`page.tsx`のstateとして`suggestResult`みたいなものを上位に持たせて、OnboardingWizardとSuggestPanelの両方から参照・更新できる形にするのが正しいかな。少し設計を整理する必要がある」

**PdM**: 「その設計は着手時に改めて考えよう。今は要件として "ウィザードで出た提案は、その後SuggestPanelを開いたときにも表示されていること" とだけ定義しておく」

**Sales**: 「技術的な話は難しいけど、ユーザーから見ると "ウィザードで答えたのに、後でAI提案ボタンを押したら空になってる" は混乱しますよね。引き継がれるべきだと思う」
