## 4. 横断分析: 6名から見えてきた共通パターン

**PdM**: 「6名のインタビューを整理すると、3つのユーザー層とそれぞれの課題が見えてきた」

### ユーザー層の整理

```
【層A】情報収集〜エリア選定中（U1・U3・U6）
  - 価格以外の情報（子育て環境・治安・路線）が不足
  - 不動産用語・表記が難しい（Q表記・㎡など）
  - "買える" という確信を持てない

【層B】特定路線・エリアで本格検討中（U2・U4）
  - 路線フィルターがなく関係ない情報が多い
  - 総コスト（リフォーム・管理費）が見えない
  - 戸建との比較ができない

【層C】データ分析・投資目的（U5）
  - 築年数別のトレンドが見たい
  - 件数が少ない駅の信頼性表示が欲しい
  - 利回り計算できる情報（賃料相場）がない
```

**DS**: 「このプロダクトの主ターゲットはどの層？ 設計によって優先する機能が変わる」

**PdM**: 「現状は層Aと層Bが混在したユーザー向けの設計になってる。層Cは副次的。ただし6名中1名（U5）がC層で、彼のフィードバックは鋭かった。投資家向け機能はコアではないにせよ、データの信頼性表示（件数が少ない駅を薄くする）は全ユーザーにとっても改善になる」

**Sales**: 「不動産購入の現場感で言うと、層Aと層Bの間の "このエリアに絞り込んで本格的に物件探しを始める段階" にいる人が一番多い。そこで背中を押せるプロダクトを作るのが良いと思う」

## 6. 機能改善提案の設計議論

### 6A. 路線フィルター（最優先）

**Dev**: 「G-17 として TODO にはあるけど "Phase 7A 待ち・低優先" になってた。今回のインタビュー結果を踏まえると優先度を上げるべきだな。`StationData` に `lines` フィールドがあるので、フィルタリング自体は難しくない」

**DS**: 「実装方針は: `lines` フィールドから全路線を一意に抽出 → FilterPanel にドロップダウン or マルチセレクトで路線選択 → 選択路線に含まれる駅のみ表示。データ側の変更は不要」

**PdM**: 「ドロップダウンより路線名を横スクロールのタブで見せる方が一覧性が高い。埼玉の主要路線は10本前後なので画面に収まる。選択なし＝全路線表示のデフォルト動作」

**Sales**: 「路線の並び順は利用者数順で。京浜東北線・埼京線・東武東上線あたりが上位のはず」

### 6B. AI 提案カードへの位置情報追加

**PdM**: 「SuggestPanel の提案カードに、現状は "駅名・エリア名・価格・おすすめ理由" しかない。ここに "○○線 / 大宮から12分" のような路線・所要時間情報を追加すると、初めて聞く駅名でも位置感がつかめる」

**Dev**: 「`StationData` に `lines` フィールドがあるので路線名は取れる。大宮・浦和・池袋など主要ターミナル駅からの所要時間は `StationData` にはないが、固定のターミナル駅リストを定義して、同一路線の駅インデックス差分で概算分数を出すことはできる。精度は粗いが "参考程度" の表示なら許容範囲」

**DS**: 「正確な所要時間は駅間距離と路線速度によるので、精度を担保したければ国交省の鉄道時刻データや外部 API が必要。まずは "○○線沿い" だけ表示して、所要時間は後から追加するフェーズ分けでいい」

### 6C. Q 表記 → 日本語化

**Dev**: 「取引一覧の "2022年第4四半期" を "2022年10〜12月" か "2022年Q4" から "2022年末" 表記に変更するだけ。`tx.period` の文字列を加工する1行修正。コスト最小・効果あり」

**PdM**: 「"2024年Q3" → "2024年7〜9月" が最も親切。Screen 2 の取引一覧で現状 `.replace("年第", "年Q").replace("四半期", "")` という処理が入っているけど、その後の変換まで追加」

### 6D. ウィザード再表示ボタン（G-10）

**Dev**: 「G-10 は "低優先" として積み残しになってたが、デモや紹介時に困るケースが出た（U3）。実装はシンプルで、FilterPanel か設定メニューに "もう一度設定する（ウィザードを再表示）" ボタンを追加して `localStorage.removeItem("onboarding_completed")` → `setShowWizard(true)` を呼ぶだけ」

**PdM**: 「ユーザー向けの機能としては "はじめからやり直す" ボタンのような体験。FilterPanel の末尾か、ヘッダーの設定アイコンに追加するのが自然」

### 6E. データ信頼性バッジ（取引件数が少ない駅）

**DS**: 「U5 の指摘通り、取引件数が10件未満の駅マーカーには "参考値" バッジを表示して半透明にする。閾値は10件で良い。`allStats.count < 10` で判定できる。信頼性の低いデータを使って意思決定するリスクを下げられる」

**Dev**: 「マーカーの opacity を 0.5 にして、ホバーツールチップに "取引件数が少ないため参考値です" を追加。実装コストは低い」

### 6F. ウィザード STEP 2 の面積マッピング柔軟化

**PdM**: 「U3 のように "1人暮らしでも広い部屋がいい" というニーズがある。現状 "👤1人 → 30㎡" のマッピングは固定。改善案: 3択を選んだ後に "面積をカスタマイズ" リンクを表示して、エリアトグルボタンで上書きできるようにする。選択した面積は STEP 3 へ引き継ぐ」

**Dev**: 「ウィザード STEP 2 の末尾に折りたたみで面積微調整 UI を追加するだけ。`targetArea` は既に `useState` で管理されているので、SetState を呼ぶだけで反映される。コスト低」

### 6G. エリア特性情報（中期）

**PdM**: 「U1 の "保育園あるか知りたい" は根本的な課題で、Phase 7C（エリアスコアリング）に対応する内容。短期では AI サマリの生成プロンプトに "子育て施設・保育園の充実度" を観点として追加することで、テキストレベルの情報補完ができる」
